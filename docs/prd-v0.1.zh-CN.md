# Pig Studio PRD v0.1（中文翻译）

## 1. 产品定义

Pig Studio 是一个基于 `pi-mono` 构建的原生跨平台桌面应用，用于创建、运行和管理 `agent sessions`（代理会话）。

产品围绕用户本地项目展开，提供一个聚焦的工作空间，用于通过持久会话、执行历史、审批和运行时设置与代理交互。

## 2. 产品定位

Pig Studio 是一个 `agent session workspace`（代理会话工作空间），不是通用 AI 聊天应用，也不是完整 IDE。

它面向希望在真实本地项目中通过跨平台桌面应用使用代理的用户。

## 3. 产品目标

1. 让用户可以打开本地项目并按项目组织工作。
2. 让用户可以创建并恢复 `agent sessions`。
3. 让用户可以在持续会话中与代理交互，并获得清晰的执行反馈。
4. 让用户可以在一个地方观察会话活动、状态和审批。
5. 提供围绕 `pi-mono` 构建的原生跨平台桌面体验。

## 4. 非目标

1. 不是多提供商 CLI 聚合器。
2. 在首个版本中不替代 IDE。
3. 不是云端编排平台。
4. 在首个版本中不是团队协作产品。
5. 不是通用聊天机器人产品。

## 5. 核心术语

### Project（项目）

用户在 Pig Studio 中打开的本地文件夹。

### Agent（代理）

由 `pi-mono` 提供并编排的一种代理能力。

### Agent Session（代理会话）

代理在某个特定项目内的持久工作会话。

### Worktree（工作树）

在 Git 项目中可选启用的隔离工作目录，用于降低会话间的代码修改冲突。Worktree 不是 v0.1 创建会话的前置条件。

### Run（运行）

用户在会话内触发的一次执行周期。

### Approval（审批）

在会话运行期间针对敏感操作发起的用户决策。

## 6. 目标用户

### 核心用户

1. 在本地代码仓库中工作的开发者。
2. 希望使用专用桌面界面进行代理工作的人。
3. 需要基于项目的会话历史和恢复能力的人。

### 用户心智

用户不是在寻找新奇聊天界面。他们需要的是一个可靠的工作空间，帮助他们快速开启代理会话、看到代理在做什么，并在之后回到未完成的工作。

## 7. 核心用户场景

1. 用户打开一个本地项目并启动新的 `agent session`。
2. 用户在该会话内发送提示并与代理迭代。
3. 会话运行时，用户查看执行进度、输出和审批请求。
4. 用户关闭应用后，稍后可从项目侧边栏恢复同一会话。
5. 用户更新 `pi-mono` 的运行时设置，并在后续会话中使用新配置。

## 8. MVP 范围

### 8.1 项目管理

1. 添加或打开本地文件夹作为项目。
2. 在左侧边栏展示项目。
3. 在本地持久化最近或置顶项目列表。

### 8.2 会话管理

1. 在项目下创建新的 `agent session`。
2. 列出每个项目下的会话。
3. 打开已有会话。
4. v0.1 会话持久化优先依赖 `pi-mono` 已有能力，Pig Studio 仅持久化必要的本地索引与界面状态。
5. 重命名和删除会话。
6. 对 Git 项目提供可选 worktree 绑定能力；对非 Git 项目保持普通目录会话流程。

### 8.3 会话交互

1. 在主区域展示对话和执行流。
2. 允许用户在活动会话中发送提示。
3. 展示会话状态，如空闲、运行中、等待审批、已完成、失败。
4. 支持会话内连续多轮交互。

### 8.4 运行时可见性

1. 按时间顺序展示执行事件。
2. 清晰展示错误。
3. 展示审批请求并允许用户响应。
4. 保留足够的运行历史用于会话恢复和调试。

### 8.5 设置

1. 在左侧边栏底部区域提供设置入口。
2. 配置 `pi-mono` 运行时位置和所需环境设置。
3. 展示运行时健康状态或可用状态。

### 8.6 平台

1. 原生桌面应用。
2. 支持 macOS、Windows、Linux 跨平台。
3. 产品以桌面端交付为唯一形态，不提供浏览器 SaaS 版本。

## 9. 功能需求

### FR-1 项目模型

1. 系统必须允许用户添加本地文件夹作为项目。
2. 系统必须在本地持久化项目元数据。
3. 系统必须在侧边栏中于项目下展示该项目的会话。
4. 系统必须同时支持非 Git 本地文件夹与 Git 仓库项目。

### FR-2 会话生命周期

1. 系统必须允许在项目上下文中创建会话。
2. 系统必须在应用重启后保留会话。
3. 系统必须支持重新打开历史会话。
4. 系统必须支持在确认后删除会话。
5. 对 Git 项目，系统应支持可选 worktree 作为会话隔离策略；该能力不可阻塞普通会话创建。

### FR-3 消息与运行

1. 系统必须允许用户向当前会话提交输入。
2. 运行进行中时，系统必须展示实时输出。
3. 系统必须将结果追加到会话历史中。

### FR-4 审批

1. 系统必须在运行期间展示审批请求。
2. 系统必须允许用户批准或拒绝请求。
3. 系统必须将审批结果记录到会话历史中。

### FR-5 持久化

1. 系统必须在本地存储项目元数据与会话索引信息。
2. v0.1 会话历史与恢复语义优先依赖 `pi-mono` 持久化能力。
3. 系统必须在重启后恢复项目与会话入口状态，并可重新进入对应会话。
4. 对于 `pi-mono` 无法恢复的会话，系统必须明确展示恢复失败并提供重建入口。

### FR-6 设置

1. 系统必须提供设置页面。
2. 系统必须校验 `pi-mono` 是否可用。
3. 系统必须以清晰方式展示配置问题。

## 10. 信息架构

### 左侧边栏

1. 项目列表。
2. 每个项目下可展开的会话列表。
3. 新建会话入口。
4. 位于侧边栏右下区域的设置按钮。

### 主区域

1. 会话头部。
2. 会话对话与事件流。
3. 输入编辑区。

## 11. 页面布局

### 左侧边栏

- 层级：`Project -> Session`
- 每个项目可展开显示其会话。
- 设置按钮位于侧边栏右下角。

### 主会话区域

- 顶部：项目名、会话名、状态。
- 主体：对话和执行历史。
- 底部：输入框、发送操作、运行状态。

## 12. UX 原则

1. 以项目为第一导航维度。
2. 强调会话连续性，而非一次性聊天。
3. 始终提供清晰执行状态。
4. 交互应原生、快速、平稳。
5. 敏感操作必须始终可见且明确。

## 13. MVP 成功标准

1. 用户可在一分钟内打开项目并创建会话。
2. 用户可重启应用并成功恢复历史会话。
3. 用户可理解会话处于运行中、阻塞、失败或等待审批。
4. 用户可在不离开应用的情况下完成所有核心会话操作。

## 14. 待定问题

1. 即便 v0.1 仅使用 `pi-mono`，未来项目是否应支持多种代理类型？
2. v0.1 需要定义哪些具体审批类别？
3. 在主会话流中应展示多少底层执行细节，哪些应放到次级检查视图？
4. 删除项目时应删除本地会话历史，还是仅将项目从应用中解除关联？

## 15. 总结

Pig Studio v0.1 是一个基于 `pi-mono` 的原生桌面 `agent session workspace`（代理会话工作空间）。

其 MVP 刻意保持聚焦：

1. 打开项目。
2. 创建并恢复代理会话。
3. 与代理对话。
4. 观察运行与审批。
5. 在简洁的原生桌面应用中管理运行时设置。
